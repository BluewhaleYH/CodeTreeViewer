# CodeTreeViewer - 에이전트 작업 가이드

## 1. 프로젝트 개요

- 사용자가 선택한 프로젝트 내부의 **파일/함수 간 관계를 관계도(태양계형) 또는 트리 형태로 시각화**하는 **네이티브 데스크톱 애플리케이션**.
- 코드 구조 탐색, 로그 덤프 -> 코드 역추적, 앱 내 코드 편집.재분석을 한 도구에서 제공하는 것이 목표.
- **주 사용자**: 안드로이드 플랫폼/프레임워크 개발자.
- **배포**: 자동 배포, 업데이트.

## 2. 기술 스택

- **Electron + TypeScript** (데스크톱 셸)
- **Tree-sitter** (코드 파싱)
- **Monaco Editor** (앱 내 코드 편집 - 2차)
- **Canvas/WebGL 기반 그래프 렌더링** (후보: Cytoscape.js / sigma.js)
- 지원 플랫폼: **Windows, Ubuntu(Linux)**
- 라이브러리는 오픈소스 라이선스(MIT/Apache-2.0 등)을 우선한다.

## 3. 문서 구조 (Single Source of Truth)

| ---- | ---- |

| `docs/02_CODE_ANALYSIS_SPEC.md` | 코드 분석 엔진: 지원 언어, 파싱, 산출물, 영역 분류, 심볼 해석, 캐시, 에러 처리, 외부 의존성 |

## 4. 개발 원칙

- **점진적 구현(작은 단위)**: 한 번에 큰 기능을 잡지 않는다. 작게 나누어 단계적으로 구현.검증한다.
- **품질 우선**: 속도보다 정확성. 청므부터 꼼꼼하고 정확하게 구현한다.
- **성공 기준은 기능별로**: 각 기능 구현 직전, 사용자와 논의하여 성공 기준을 확정한 뒤 진행한다. (`00` §9)

### 단계별 범위 (`00` §8.1 기준)

- **MVP(1차)**: Java/Kotlin 파일 의존석 분석, 관계도/트리 시각화, 프로젝트 폴더 선택, 다중 탭, 탭/열린 파일/뷰 모드 상태 저장, 파일/함수명 기본 검색.
- **2차**: 함수 호출 그래프, 로그 덤프 분석, 앱 내 코드 편집/저장/재분석, C/C++ native 파일 의존성 분석.
- **추후**: JNI 경계 분석 고도화, 로그 매칭 신뢰도 표시, 그래프 전/후 비교 모드, 라이트 모드.
- 단계가 명시된 기능은 **현재 단계 범위를 벗어나 미리 구현하지 않는다.** 확장이 필요하면 먼저 사용자와 단계를 합의한다.

## 5. 세션 시작 절차 (반드시 먼저 수행)

세션이 시작되면 **다른 작업에 착수하기 전에** 아래를 순서대로 먼저 수행한다.

1. **`HANDOFF.md` 읽기** - 이전까지의 작업을 복기한다. (어디까지 했고, 무엇이 진행 중이며, 다음 할 일이 무엇인지)
2. **`docs/TODO.md` 확인** - 미완료 체크박스를 위에서부터 확인하고, **순차적으로** 다음 작업을 진행한다. 임의로 순서를 건너뛰지 않는다.

## 6. 투두 구현 작업 워크플로 (체크박스 단위, 필수)

`docs/TODO.md`의 **모든 구현 작업은 체크박스 1개를 단위**로 아래 절차를 반드시 따른다. 한 번에 하나의 체크박스만 진행하고, 그 체크박스가 main에 머지된 뒤 다음으로 넘어간다.

1. **스펙 문서 읽기 (가장 먼저)**: 브랜치를 만들기 전에, 해당 체크박스에 링크된 스펙 문서(및 라인)를 **반드시 먼저 읽는다.**
2. **브랜치 생성**: 연동된 git에서 **새 브랜치**를 따고, 그 브랜치에서만 작업한다. (main에서 직접 작업 금지)
3. **구현**.
4. **자체 검수**: 구현 후 자체 검수한다.
   - **눈에 보이는 결과(UI 등)가 있는 체크박스라면**, 프로그램을 실행해 **스크린샷을 캡처**해 확인한다. 스크린샷은 **로컬 `screenshots/` 폴더에만 저장**하며 **git에는 올리지 않는다.** (`screenshots/`는 `.gitignore`에 포함 - 불필요한 이미지 커밋 방지)
   - 헤드리스 환경에서 캡처가 필요하면 가상 디스플레이(`xvfb` 등)를 사용한다.
5. **리뷰 요청 파일 생성**: 자체 검수가 끝나면 `reviews/unprocessed/M<마일스톤>_<번호>.md`를 생성해 리뷰를 진행한다(파일명은 체크박스에 대응, 예: M5 세 번째 -> `M5_3.md`). 내용에는 **구현 요약, 관련 스펙(`파일 §섹션`), 브랜치명, 주요 변경 파일**을 적는다.
6. **심화/세부 리뷰**: §6.1에 따라 `reviews/unprocessed/`의 요청 파일을 읽고 리뷰한다. 리뷰 결과는 `reviews/M<n>_<k>.md`(최상위)에 기록한다. (반드시 자체 검수 -> 리뷰) 이후 §6.2에 따라 결과를 반영한다.
7. **푸시 & 머지 리퀘스트**: 리뷰 반영까지 끝나면 git에 push하고 **머지 리퀘스트(MR/PR)**를 작성한다.
8. **머지**: 머지는 main 브랜치로 머지한다.
9. **마무리**: 머지 후 `docs/TODO.md`의 해당 체크박스를 체크한다. (`HANDOFF.md` 작성/갱신은 사용자 지시에 따라 별도로 수행)

### 6.1 리뷰 요청 처리 (`reviews/unprocessed/`)

- 먼저 **`reviews/unprocessed/`** 안의 리뷰 요청 파일을 확인한다.
- 요청 파일명은 구현한 TODO 번호와 동일하다. 예: `M5_3.md`는 `docs/TODO.md`의 `M5` 섹션 세 번째 체크박스 작업을 뜻한다.
- 파일명으로 대상 TODO를 식별한 뒤, `docs/TODO.md`의 해당 체크박스와 연결된 SPEC 문서를 먼저 읽는다.
- 구현이 해당 TODO/SPEC 목적에 맞는지, 버그 가능성(NPE/null/undefined/예외), UI 구성, UX 깨짐, 테스트/빌드 결과를 검토한다.
- 리뷰 결과는 반드시 아래 형식으로 작성하고, **`reviews/M<마일스톤>_<번호>.md`** 에 저장한다.

```text
검토 TODO : {TODO 내용}

Findings:
...

Assumptions:
...

Verification:
...
```

- `reviews/unprocessed/`의 요청 파일을 이동하지 않는다. 리뷰 결과 파일만 `reviews/` 최상위에 작성한다.
- `reviews/`는 `.gitignore` 대상(로컬 전용)이다.

### 6.2 리뷰 결과 픽업 (`reviews/`)

- 리뷰 결과는 **`reviews/M<마일스톤>_<번호>.md`** 형태로 둔다(체크박스에 대응, 예: M5 세 번째 -> `reviews/M5_3.md`).
- 해당 파일을 **Read**해서 검토하고, §6대로 필수 수정을 반영한다.
- 처리 완료한 리뷰 결과 파일은 **`reviews/processed/`로 이동**해 재감지를 막는다.

## 7. 작업 방식 / 협업 규칙

- 작업은 **한 단위씩 끊어서** 진행한다. 한 작업비 끝나면 다음 단계로 곧장 밀고 나가지 말고, 현재 단계를 깔끔히 마무리하고 산출물 위치를 알린 뒤 사용자의 다음 지시를 기다린다.

## 8. 문서 작성 규칙

- 문서 작성은 **한국어**.
- 각 SPEC 문서 최하단의 **`추가)`** 섹션은 "에이전트가 제안하는 보완/미해결 항목" 자리다. 결정이 끝난 항목은 본문(또는 `결정 사항`)으로 올리고, `추가)`에는 미해결/추후 검토 항목만 남긴다.
- **1차/2차/추후"는 프로젝트 단계 용어**다. 기능 내부의 우선순위를 가리킬 때는 이 단어를 피하고 "우선/기본" 등 다른 표현을 쓴다.
- 문서 간 참조는 `파일명 §섹션` 형식으로 적고, 섹션 번호가 바뀌면 **참조하는 모든 문서를 함께 갱신**한다. (`docs/TODO.md`의 라인 참조도 함께 점검)

## 9. 빌드 / 실행 / 테스트

- **의존성 설치**: `npm install`
- **개발 실행**: `npm run dev` (electron-vite, HMR)
  - 리눅스에서 `chrome-sandbox` SUID 미설정으로 막히는 것을 피하려고 `dev`/`start`는 `cross-env ELECTRON_DISABLE_SANDBOX=1`로 개발 실행 시에만 샌드박스를 끈다. 패키징 배포본(AppImage/deb)은 정상 샌드박스로 동작한다.
- **검사**: `npm run typechekc` / `npm run lint` / `npm run format`
- **프로덕션 빌드**: `npm run build` (-> `out/`)
- **패키징** (-> `dist/`, gitignore 대상). 자동 배포.무서명.자동업데이트(01 §9). 상세 : `docs/DEPLOY.md`:
  - Linux: `npm run build:linux` -> AppImage + deb
  - Windows 포터블 zip: `npm run build:win` (wine 크로스 빌드 가능)
  - Windows nsis 설치본 + zip: `npm run build:win:installer` (**Windows 네이티브/CI 전용** - nsis는 wine 크로스빌드 불가)
- **테스트**: `npm test` (Vitest, 1회 실행) / `npm run test:watch` (워치). 테스트는 `test/`에 둔다. (현재 스코므 테스트만 - 실제 테스트는 M1+ 기능과 함께 추가)
