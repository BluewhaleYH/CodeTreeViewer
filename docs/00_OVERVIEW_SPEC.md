# 00. 개요 (OVERVIEW) SPEC

> 본 문서는 CodeTreeViewer 기획·요구사항의 **단일 출처(Single Source of Truth)** 중 최상위 문서다.
> 전체 개요·용어·기능 맵·개발 원칙·단계별 범위·성공 기준·결정 사항을 담는다.
> 세부 기능은 `01`~`06` SPEC으로 분리한다.

## 1. 목적과 배경

- 사용자가 선택한 프로젝트 내부의 **파일/함수 간 관계를 시각적으로 탐색**하게 해 주는 **네이티브 데스크톱 애플리케이션**이다.
- 관계를 **관계도(태양계형)** 또는 **트리** 형태로 표현하여, 대규모 코드베이스의 구조를 직관적으로 파악하게 한다.
- 더불어 **로그 덤프 → 코드 역추적**, **앱 내 코드 편집·재분석**을 하나의 도구에서 제공한다.
- **주 사용자**: 안드로이드 플랫폼/프레임워크 개발자. (대규모 멀티-언어 코드베이스, 로그 기반 디버깅이 일상인 사용자)

## 2. 용어 정의

| 용어 | 정의 |
| ---- | ---- |
| **노드(Node)** | 그래프 상의 단위. 파일 노드 또는 함수 노드. (`02` §4) |
| **엣지(Edge)** | 노드 간 관계. 파일 의존성 또는 함수 호출 관계. (`02` §4) |
| **관계도(태양계형)** | 선택 노드를 중심에 두고 부모/자식 관계를 방사형으로 배치하는 뷰. (`03` §3) |
| **트리** | 계층 구조를 위→아래(또는 좌→우)로 펼치는 뷰. (`03` §4) |
| **영역(Domain/Zone)** | 코드가 속한 논리 계층(예: Application/Framework/Stack). 색으로 구분. (`02` §5, `03` §6) |
| **부모/자식** | 어떤 노드를 **참조하는 쪽**(부모)과 **참조되는 쪽**(자식). 방향성은 `02` §4에서 확정. |
| **호출처(Caller)** | 특정 함수를 호출하는 위치. 함수 노드 클릭 시 역방향 추적 대상. (`02` §6, `03` §5) |
| **세션(Session)** | 앱 종료 후에도 복원되는 상태 묶음(탭, 열린 파일, 뷰 모드 등). (`01` §5) |
| **프로젝트(Project)** | 사용자가 선택한 분석 대상 루트 폴더. 탭 1개 = 프로젝트 1개. (`01` §4) |

## 3. 전체 기능 맵

| # | 기능군 | 요약 | 상세 문서 |
| - | ------ | ---- | --------- |
| A | 앱 환경 | 네이티브 셸, 창 제어, 다중 탭, 상태 영속성, 단일 인스턴스, 자동 업데이트 | `01_APP_ENVIRONMENT_SPEC.md` |
| B | 코드 분석 | 언어 파싱, 파일/함수 관계 산출, 영역 분류, 심볼 해석, 캐시 | `02_CODE_ANALYSIS_SPEC.md` |
| C | 시각화 | 관계도/트리, 뷰 전환, 인터랙션, 색상, 대규모 렌더링, 그래프↔코드 동기화 | `03_VISUALIZATION_SPEC.md` |
| D | 로그 분석 | 로그 덤프 열람, 로그→코드 역추적, 3-뷰 연동, 필터링 | `04_LOG_ANALYSIS_SPEC.md` |

| F | 코드 편집 | 앱 내 편집, 저장, 재분석 트리거, diff, 영향 범위 | `06_CODE_EDITING_SPEC.md` |

## 4. 기술 스택

CLAUDE.md §2와 동기화한다. (변경 시 양쪽을 함께 갱신)

- **Electron + TypeScript** — 데스크톱 셸
- **Tree-sitter** — 코드 파싱 (`02`)
- **Monaco Editor** — 앱 내 코드 편집 (2차, `06`)
- **그래프 렌더링** — Canvas/WebGL 기반. 후보: Cytoscape.js / sigma.js (`03` §7, 미확정)
- 라이브러리는 **오픈소스 라이선스(MIT/Apache-2.0 등)** 를 우선한다.

## 5. 지원 플랫폼

- **Windows**, **Ubuntu(Linux)**.
- 브라우저 기반이 아닌 **독립 실행 프로그램**이어야 한다. (`01` §2)
- 배포·업데이트는 자동. (상세: `01` §9, `docs/DEPLOY.md`)

## 6. 비기능 요구사항(개요)

| 항목 | 방향 |
| ---- | ---- |
| 정확성 | 속도보다 정확성 우선. 잘못된 관계를 그리지 않는 것을 최우선. (§8) |
| 성능 | 대규모 코드베이스(수천~수만 노드)에서도 탐색 가능해야 함. 구체 기준은 기능별 성공 기준에서 확정. (`03` §8) |

## 7. 개발 원칙

CLAUDE.md §4와 동기화한다.

- **점진적 구현(작은 단위)**: 한 번에 큰 기능을 잡지 않고, 작게 나누어 단계적으로 구현·검증한다. (구현 단위는 `docs/TODO.md`의 체크박스)
- **품질 우선**: 속도보다 정확성. 처음부터 꼼꼼하고 정확하게 구현한다.
- **성공 기준은 기능별로 확정**: 각 기능 구현 직전, 사용자와 논의하여 성공 기준을 확정한 뒤 진행한다. (§9)

## 8. 단계별 범위 (§8.1)

> 단계가 명시된 기능은 **현재 단계 범위를 벗어나 미리 구현하지 않는다.** 확장이 필요하면 먼저 사용자와 단계를 합의한다.

### 8.1 단계 정의

- **MVP(1차)**
  - Java/Kotlin **파일 의존성** 분석 (`02`)
  - **관계도/트리 시각화**, 뷰 전환 (`03`)
  - **프로젝트 폴더 선택**, **다중 탭** (`01`)
  - **탭/열린 파일/뷰 모드 상태 저장** (`01` §5)
  - **파일/함수명 기본 검색** (`05`)
  - **영역 색상 구분** (`03` §6 / `02` §5)

- **2차**
  - **함수 호출 그래프** (`02` §6, `03` §5)
  - **로그 덤프 분석** (`04`)
  - **앱 내 코드 편집/저장/재분석** (`06`)
  - **C/C++ native 파일 의존성 분석** (`02`)

- **추후**
  - JNI 경계 분석 고도화 (`02`)
  - 로그 매칭 신뢰도 표시 (`04`)
  - 그래프 전/후 비교 모드 (`03`)
  - 라이트 모드 (`03` §9)

## 9. 성공 기준

- 기능별 성공 기준은 **해당 기능 구현 직전에 사용자와 확정**한다. (§7)
- 확정된 성공 기준은 각 SPEC 문서의 관련 섹션 또는 `docs/TODO.md` 해당 체크박스 옆에 명시한다.
- 본 문서에는 전 기능 공통 합격선만 둔다:
  - **C1. 정확성**: 시각화/역추적 결과가 실제 코드 구조와 일치한다. (오탐을 그리지 않는다)
  - **C2. 복원성**: 정상/비정상 종료 후 재시작 시 마지막 세션이 복원된다. (`01` §5, §10)
  - **C3. 이식성**: Windows/Linux에서 동일하게 동작한다.

## 10. 결정 사항

> 논의가 끝나 확정된 항목만 기록한다. (미해결은 `추가)`)

- **D1.** 배포 형태는 네이티브 데스크톱 앱(Electron). 브라우저 형태 아님.
- **D2.** MVP 분석 대상 언어는 Java/Kotlin, 분석 종류는 **파일 의존성**.
- **D3.** 기본 테마는 다크. 라이트 모드는 추후. (`03` §9)
- **D4.** 라이브러리는 OSS 친화 라이선스(MIT/Apache-2.0 등) 우선.
- **D5.** 부모/자식 방향: **의존하는 쪽(참조하는 쪽)=부모, 의존되는 쪽=자식**. (`02` §4.3)
- **D6.** 영역(Domain) 분류 방식: **사용자 정의 매핑 + 안드로이드 기본 프리셋**. (`02` §5)
- **D7.** MVP에서 함수/메서드 **정의**를 추출해 이름 검색·라벨에 사용. 함수 **호출 그래프/엣지**는 2차. (`02` §1·§4.1, `05` §2)
- **D8.** 코드의 구조·관계 **시각화는 관계도/트리 형태로만** 표현한다. 막대/원형 등 다른 차트 형식은 쓰지 않는다(로그/코드 텍스트 패널은 보조 영역으로 제외). (TEMP_SPEC, `03` §2)
- **D9.** import가 프로젝트 밖(SDK/외부 라이브러리)으로 해석되면 **`external` 노드로 표시**하고 표시 on/off 토글을 제공한다. (`02` §9, §4.1)
- **D10.** 영역(Domain) 분류 프리셋은 특정 레이아웃(AOSP 등)을 가정하지 않고 **모듈/최상위 디렉터리 기반 자동 그룹 + 사용자 매핑(범용)**. (`02` §5)
- **D11.** 분석 실행은 **메인 프로세스 협력적 양보(비차단)**. 진짜 워커 스레드는 추후 최적화. (`02` §8, 추가-6)
- **D12.** 앱 UI 표시 언어는 **한국어**. (`01`)
- **D13.** 단축키 수정자 키는 **Ctrl 기준**(주 대상 Windows/Linux, macOS는 배포 대상 아닌 개발 환경). (`01` §8)
