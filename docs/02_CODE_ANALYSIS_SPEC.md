# 02. 코드 분석 (CODE ANALYSIS) SPEC

> 코드 분석 엔진: 지원 언어, 파싱, 산출물(노드/엣지 모델), 영역 분류, 심볼 해석, 캐시, 에러 처리, 외부 의존성을 정의한다.
> 시각화는 `03`, 검색은 `05`, 편집 후 재분석은 `06`.

## 1. 범위 / 단계

- **MVP**:
  - Java/Kotlin **파일 의존성** 분석.
  - Java/Kotlin **함수/메서드 정의 추출** — 이름 검색(`05` §2)·노드 라벨용. (호출 관계는 만들지 않음)
- **2차**: **함수 호출 그래프**(§6, 호출 엣지), **C/C++ native 파일 의존성**.
- **추후**: JNI 경계 분석 고도화.
- 현재 단계를 벗어나 미리 구현하지 않는다. (`00` §8)
- 참고: MVP에서 함수 **정의**는 추출하지만(검색·라벨), 함수 **호출 그래프/엣지**는 2차다. 둘을 혼동하지 않는다.

## 2. 지원 언어

| 언어 | 분석 종류 | 단계 |
| ---- | -------- | ---- |
| Java | 파일 의존성 | MVP |
| Kotlin | 파일 의존성 | MVP |
| Java/Kotlin | 함수/메서드 정의 추출(검색·라벨용) | MVP |
| Java/Kotlin | 함수 호출 그래프(호출 엣지) | 2차 |
| C/C++ | 파일 의존성 | 2차 |
| JNI 경계 | 경계 매핑 | 추후 |

## 3. 파싱

- **Tree-sitter**를 사용한다. (`00` §4) 언어별 문법: tree-sitter-java, tree-sitter-kotlin, (2차) tree-sitter-c/cpp.
- 파싱 단위: 파일. 프로젝트 폴더를 재귀 스캔하여 대상 확장자 파일을 수집한다.
- 파싱 실패 파일은 **건너뛰되 기록**하고 분석을 계속한다(부분 결과 허용). (§8)
- 증분 분석: 편집·재분석 시 변경 파일만 다시 파싱한다. (`06` 연동, 2차)

## 4. 산출물 — 노드/엣지 모델

분석 결과는 **그래프 모델**(노드 + 엣지)로 산출하며 `03`이 이를 렌더한다.

### 4.1 노드(Node)

- 종류: `file`(MVP), `function`(정의는 MVP, 호출 관계는 2차).
  - `function` 노드의 **정의는 MVP에서 생성**한다(이름 검색·라벨·소스 이동용).
  - `function` 노드 사이의 **호출 엣지(§4.2 `function-call`)는 2차**다. (MVP에서는 함수 간 연결선을 그리지 않음)
- 공통 필드(초안): `id`, `kind`, `name`, `path`, `language`, `domain`(영역, §5), 위치(라인) 등.

### 4.2 엣지(Edge)

- 종류:
  - `file-dependency` (MVP): 파일 A가 파일 B를 의존(import/include 등).
  - `function-call` (2차): 함수 A가 함수 B를 호출.
- 필드(초안): `from`, `to`, `type`, 출처 위치(라인).

### 4.3 방향성 정의 (부모/자식) — **확정**

- **부모 = 의존하는 쪽(참조하는 쪽)**, **자식 = 의존되는 쪽(참조되는 쪽)** 으로 **확정**한다. (`00` §10 D5)
  - 예: `A.java`가 `B.java`를 import → 엣지 `A → B`. A가 부모, B가 자식.
  - 즉 진입점에서 하위 의존으로 내려가는 방향이 부모 → 자식이다.
- 특정 파일 클릭 시 그 파일을 중심으로 **부모(자신을 참조하는 파일)와 자식(자신이 참조하는 파일)** 을 함께 표시한다. (TEMP_SPEC, `03` §5)
- 특정 함수 클릭 시 그 함수를 **마지막 노드로 하여 호출하는 곳들(호출처)** 을 역방향으로 표시한다. (TEMP_SPEC, §6, `03` §5)

## 5. 영역(Domain/Zone) 분류

- 코드가 속한 논리 영역을 분류하고, 시각화에서 색으로 구분한다. (TEMP_SPEC, `03` §6)
  - 예시(안드로이드): Application=주황, Framework=초록, Stack=파랑.
- **분류 방식(확정, `00` §10 D6·D10)**: **사용자 정의 매핑 + 범용 기본 프리셋**.
  - 사용자가 **경로/모듈/패키지 패턴 → 영역** 매핑을 정의할 수 있다.
  - 기본 프리셋은 특정 레이아웃(AOSP 등)을 가정하지 않고 **모듈/최상위 디렉터리 기반 자동 그룹**으로 영역을 부여한다(무설정 동작). AOSP 레이어/Gradle 모듈 모두에 무난히 적용.
  - 매핑 규칙은 우선순위(구체적 패턴 우선)로 평가하고, 매핑은 프로젝트(탭) 단위로 저장한다.
- 분류 불가 노드는 `unknown` 영역(중립색)으로 둔다.
- 프리셋의 구체 패턴 정의는 `추가)`에서 상세화한다.

## 6. 함수 호출 그래프 (2차)

- 함수 **정의**는 MVP에서 이미 추출된다(§4.1). 본 절은 **호출 관계(`function-call` 엣지) 구축**을 다루며 **2차** 범위다.
- 추출된 함수 정의를 기반으로 함수 간 호출을 해석하여 `function-call` 엣지를 만든다.
- 함수 클릭 시 **해당 함수를 호출하는 경로(호출처 체인)** 를 역추적해 표시한다. (TEMP_SPEC)
- 정확한 호출 해석을 위해 심볼 해석(§7)이 필요하다.

## 7. 심볼 해석 / 캐시

### 7.1 심볼 해석

- import/패키지/클래스 기준으로 참조 대상을 실제 파일/심볼에 연결한다.
- 해석 실패(외부 라이브러리, 미발견 심볼)는 **외부/미해결 노드**로 분리 표시하고 오탐을 만들지 않는다. (`00` §9 C1)

### 7.2 캐시

- 분석 결과는 캐시하여 재기동·재방문 시 재사용한다.
- 무효화 키: 파일 경로 + 내용 해시(또는 mtime+size) + 분석기 버전.
- 저장 위치는 세션 상태와 분리한다(`01` §6). 사용자 프로젝트 폴더는 오염하지 않는다.

## 8. 에러 처리

- 파싱/해석 실패는 분석 전체를 중단시키지 않는다(부분 결과 + 실패 목록).
- 실패 항목은 사용자에게 비차단 형태로 요약 제공(개수/목록).
- 권한 없는 파일·심볼릭 링크 루프 등은 안전하게 건너뛴다.

## 9. 외부 의존성

- 프로젝트 외부(SDK/라이브러리) 심볼은 **외부 노드**로 구분하여 그래프를 오염시키지 않는다.
- 외부 노드 표시 여부는 시각화 옵션으로 토글 가능(설계: `03` `추가)`).

## 10. 성공 기준(초안)

- **B1.** Java/Kotlin 프로젝트에서 파일 의존성(import 기반)이 실제 코드와 일치하게 추출된다.
- **B2.** 해석 불가 참조를 자식으로 오탐하지 않는다(외부/미해결로 분리). (`00` §9 C1)
- **B3.** 대규모 프로젝트에서 분석이 완료되며 부분 실패가 있어도 결과를 제공한다.

> 확정 전 사용자와 합의. (`00` §9)
